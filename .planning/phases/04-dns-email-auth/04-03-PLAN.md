  <action>
    Create the dns/validator/ package for DNS record validation:

    **checker.go** - DNS record validation (per user decision: validation built into dns-setup + --validate-only for standalone):
    - `type CheckResult struct { RecordType, Name, Expected, Actual string; Pass bool; Error string }` - result for a single record check.
    - `type ValidationReport struct { Results []CheckResult; AllPassed bool; Timestamp time.Time }`
    - `type Checker struct` - holds DNS client (miekg/dns), configurable DNS servers.
    - `NewChecker(servers []string) *Checker` - creates checker querying specified DNS servers (default: 8.8.8.8:53, 1.1.1.1:53).
    - `CheckSPF(ctx context.Context, domain string, expectedIP string) CheckResult` - query TXT records for domain, find one starting with "v=spf1", verify it contains `ip4:{expectedIP}`. Fail if no SPF record found, fail if multiple SPF records (pitfall #8), fail if expected IP not included.
    - `CheckDKIM(ctx context.Context, domain string, selector string) CheckResult` - query TXT record at `{selector}._domainkey.{domain}`, verify starts with "v=DKIM1". Check key type (k=rsa) and that p= contains a non-empty base64 value.
    - `CheckDMARC(ctx context.Context, domain string) CheckResult` - query TXT record at `_dmarc.{domain}`, verify starts with "v=DMARC1", report p= policy value.
    - `CheckMX(ctx context.Context, domain string, expectedHostname string) CheckResult` - query MX records, verify at least one MX pointing to expectedHostname.
    - `CheckAll(ctx context.Context, domain, expectedIP, relayHostname, dkimSelector string) ValidationReport` - runs all checks, aggregates results.

    Use miekg/dns for all queries (not stdlib net.LookupTXT) per research: need controlled DNS server selection, raw record parsing, and consistent behavior across environments.

    **ptr.go** - PTR reverse DNS verification (per user decision: verify PTR, automate where possible):
    - `type PTRResult struct { IP, ExpectedHostname string; PTRNames []string; ForwardMatch bool; Pass bool; Error string }`
    - `CheckPTR(ctx context.Context, ip string, expectedHostname string) PTRResult` - performs:
      1. Reverse DNS lookup: IP -> hostnames (using net.LookupAddr which handles in-addr.arpa construction)
      2. For each returned hostname, forward DNS lookup: hostname -> IPs
      3. Verify original IP appears in forward lookup results
      4. Verify expectedHostname appears in PTR names
      This implements the RFC-standard PTR check. Use stdlib net package for PTR (it handles the reverse zone construction correctly). Alternative: use mjl-/mox/iprev if stdlib is insufficient, but try stdlib first to minimize dependencies.
    - Document in PTRResult.Error when PTR is missing: explain that PTR records are set by the VPS provider, not the DNS provider, and link to Phase 1 VPS provider guide for instructions.

    Tests should cover:
    - SPF check passes when ip4: mechanism contains expected IP
    - SPF check fails when no SPF record exists
    - SPF check detects multiple SPF records (pitfall #8)
    - DKIM check passes when selector._domainkey has valid v=DKIM1 record
    - DKIM check fails when selector not found
    - DMARC check passes for valid v=DMARC1 record
    - MX check passes when expected hostname is in MX records
    - PTR check passes when reverse and forward DNS match
    - PTR check fails when no PTR record exists (with helpful error message)
    - CheckAll aggregates all results correctly

    Use a mock DNS server in tests (miekg/dns can serve as a test DNS server) to avoid live DNS queries in unit tests.

    - `type AuthResult struct { Method string; Result string; Details string }` - e.g., Method="dkim", Result="pass", Details="header.s=darkpipe-2026q1".
    - `type AuthReport struct { Results []AuthResult; SPFPass, DKIMPass, DMARCPass bool }`
    - `ParseAuthResults(header string) (AuthReport, error)` - parses Authentication-Results header from email. Use emersion/go-msgauth/authres for RFC 8601 compliant parsing (per research: don't hand-roll regex for this). Extract DKIM, SPF, and DMARC results. Set convenience booleans (SPFPass, DKIMPass, DMARCPass).

    **authtest/sender.go** - Test email sender for end-to-end verification:
    - `type TestEmailConfig struct { From, To, RelayHost string; RelayPort int; DKIMSigner *dkim.Signer }`
    - `SendTestEmail(ctx context.Context, cfg TestEmailConfig) error` - constructs a minimal test email with From, To, Subject ("DarkPipe DNS Authentication Test"), Date, Message-ID headers. Signs with DKIM via the provided Signer. Sends via the cloud relay SMTP (RelayHost:RelayPort) using emersion/go-smtp client. Include instructions in output: "Check the received email headers for Authentication-Results to verify SPF, DKIM, and DMARC pass."
    - This is a helper for manual end-to-end testing. Users run it after DNS setup to verify authentication works.

    **cmd/dns-setup/main.go** - Unified CLI entry point (per user decision: single `darkpipe dns-setup` command):
    - Parse command-line flags:
      - `--domain` (string, required) - domain to configure
      - `--relay-hostname` (string, required) - cloud relay FQDN
      - `--relay-ip` (string, required) - cloud relay public IP
      - `--apply` (bool, default false) - actually create DNS records (dry-run by default per user decision)
      - `--validate-only` (bool) - only validate existing DNS records, don't generate/create anything
      - `--rotate-dkim` (bool) - rotate DKIM key (generate new selector, publish, switch)
      - `--json` (bool) - JSON output mode (per user decision)
      - `--dkim-key-dir` (string, default /etc/darkpipe/dkim) - DKIM key storage location
      - `--dkim-selector-prefix` (string, default "darkpipe") - selector prefix
      - `--dmarc-policy` (string, default "none") - DMARC policy (none/quarantine/reject)
      - `--send-test` (string) - send test email to this address after setup
    - Also read from environment variables (flags override env vars).

    - **Workflow (default mode: dns-setup --domain example.com --relay-hostname mail.example.com --relay-ip 1.2.3.4):**
      1. Load config from flags + env vars
      2. Generate DKIM key pair if not exists (or load existing). Save to --dkim-key-dir.
      3. Generate all DNS records (SPF, DKIM, DMARC, MX) using dns/records/
      4. Auto-detect DNS provider via NS lookup using dns/provider/detector
      5. If supported provider (Cloudflare/Route53):
         - DRY RUN (default): Show what records would be created
         - WITH --apply: Create records via API, wait for propagation, then validate
      6. If unsupported provider:
         - Print records to terminal (colored checklist)
         - Save DNS-RECORDS.md guide file
      7. Run validation checker against live DNS (per user decision: validation runs after apply)
      8. If --send-test provided: send test email and print instructions for checking headers

    - **Workflow (--validate-only mode):**
      1. Load config
      2. Run all DNS validation checks (SPF, DKIM, DMARC, MX, PTR)
      3. Print results as colored checklist or JSON

    - **Workflow (--rotate-dkim mode):**
      1. Generate new key pair with next quarter's selector
      2. Publish new DKIM TXT record (via API or print for manual)
      3. Wait for propagation
      4. Print instructions: "Update mail server to sign with new selector. Keep old selector DNS record for 7 days."

    - Output formatting (per user decision):
      - Default: human-friendly colored checklist with pass/fail indicators
      - --json: structured JSON with all record values and validation results
      - Always print to terminal AND save DNS-RECORDS.md for manual reference

    CLI should be buildable as: `go build -o darkpipe-dns ./dns/cmd/dns-setup/`

    Tests for parser:
    - Parse Gmail-style Authentication-Results header
    - Parse header with all three methods (DKIM, SPF, DMARC) passing
    - Parse header with mixed results (SPF pass, DKIM fail)
    - Handle malformed headers gracefully (return error, not panic)
