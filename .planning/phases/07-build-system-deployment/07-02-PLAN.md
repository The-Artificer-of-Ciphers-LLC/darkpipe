    type Config struct {
        Version       string `yaml:"version"`        // Schema version for migration

        RelayHostname string `yaml:"relay_hostname"`  // Cloud relay FQDN (e.g., relay.example.com)

    **1. deploy/setup/cmd/darkpipe-setup/main.go:**
    Create Cobra CLI with two commands:
    - `setup` (default command) -- Interactive setup wizard
    - `version` -- Print version info

    **Setup command flow:**

    a) **Detect existing config:** Check for `.darkpipe.yml` in current directory.
       - If found: load config, check if migration needed, ask "Existing config found (version X). Upgrade? [Y/n]"
       - If upgrade: run migrations, then present current values as defaults in prompts
       - If not found: use DefaultConfig() as starting defaults

    b) **Quick vs Advanced mode (UX-01 tiered experience):**
       Ask: "Setup mode: [Quick (recommended defaults)] / [Advanced (customize everything)]"
       - Quick: Only ask for mail_domain, relay_hostname, admin_email. Use defaults for everything else (Stalwart, SnappyMail, builtin calendar, WireGuard, queue enabled).
       - Advanced: Ask all questions below.

    c) **Interactive prompts with live validation (using survey.v2):**
       1. **Mail domain** (Input) -- validate with `validate.ValidateDomain()`, show spinner during DNS lookup. If DNS fails, warn but allow continuation (user may not have set up DNS yet).
       2. **Relay hostname** (Input) -- validate with `validate.ValidateSMTPPort()` for port 25. Warn if port 25 closed (common VPS issue).
       3. **Admin email** (Input) -- basic format validation
       4. **Mail server** (Select: stalwart/maddy/postfix-dovecot, default: stalwart) -- Show brief description of each
       5. **Webmail** (Select: none/roundcube/snappymail, default: snappymail) -- if mail_server is stalwart and calendar is builtin, note SnappyMail is recommended for lighter footprint
       6. **Calendar/Contacts** (Select: none/radicale/builtin, default: builtin) -- if mail_server != stalwart, filter out "builtin" option (Stalwart-only). Auto-default to radicale for maddy/postfix-dovecot.
       7. **Transport** (Select: wireguard/mtls, default: wireguard) -- Advanced only
       8. **Queue enabled** (Confirm, default: true) -- Advanced only
       9. **Strict TLS mode** (Confirm, default: false) -- Advanced only
       10. **Extra domains** (MultilineInput, optional) -- Advanced only

    d) **Validation summary:** Use pterm.DefaultTable to display all selected options. Ask "Proceed with this configuration? [Y/n]"

    e) **Generate outputs:**
       1. Generate Docker secrets (call secrets.GenerateSecrets)
       2. Generate docker-compose.yml (call compose.Generate)
       3. Save config to .darkpipe.yml (call config.SaveConfig)
       4. Create `.darkpipe-configured` marker file
       5. Use pterm progress bar for each step

    f) **Success output:** Use pterm.Success to show completion, then print next steps:
       ```
       Next steps:
       1. Review docker-compose.yml
       2. Set up DNS records: darkpipe dns-setup --domain example.com
       3. Start services: docker compose up -d
       4. Check logs: docker compose logs -f
       ```

    **2. deploy/setup/pkg/compose/generate.go:**
    - `Generate(config *Config, outputPath string) error` -- Generate docker-compose.yml from config
    - Build compose structure programmatically (NOT string templates -- use gopkg.in/yaml.v3 for type-safe YAML generation)
    - Define service structs that map to Docker Compose spec:
      ```go
      type ComposeFile struct {
          Version  string                    `yaml:"version"`
          Services map[string]*ComposeService `yaml:"services"`
          Networks map[string]*ComposeNetwork `yaml:"networks"`
          Volumes  map[string]*ComposeVolume  `yaml:"volumes,omitempty"`
          Secrets  map[string]*ComposeSecret  `yaml:"secrets,omitempty"`
      }
      ```
    - Always include: cloud-relay service (from existing docker-compose), darkpipe network
    - Conditionally include based on config:
      - Mail server: stalwart OR maddy OR postfix-dovecot (one of three)
      - Webmail: roundcube OR snappymail OR none
      - Calendar: radicale OR none (builtin handled by stalwart itself, no extra service)
      - Shared services: rspamd + redis (always included, needed by all mail servers)
      - Caddy reverse proxy (always included for webmail/CalDAV HTTPS)
    - Add Docker secrets block for sensitive values:
      ```yaml
      secrets:
        admin_password:
          file: ./secrets/admin_password.txt
        dkim_private_key:
          file: ./secrets/dkim_private_key.pem
      ```
    - Reference secrets in services that need them:
      ```yaml
      services:
        stalwart:
          secrets:
            - admin_password
          environment:
            ADMIN_PASSWORD_FILE: /run/secrets/admin_password
      ```
    - **GHCR image references:** All service image fields MUST use `ghcr.io/trek-e/darkpipe/` prefix. Specifically:
      - Cloud relay: `ghcr.io/trek-e/darkpipe/cloud-relay:${VERSION:-latest}`
      - Home mail server: `ghcr.io/trek-e/darkpipe/home-${mail_server}:${VERSION:-latest}` (where mail_server is stalwart, maddy, or postfix-dovecot)
      - This ensures the generated compose file references the same GHCR images that Plan 07-01 workflows publish. Never use Docker Hub image paths.
    - Use environment variables for non-sensitive config (MAIL_DOMAIN, RELAY_HOSTNAME, etc.)
    - Set appropriate memory limits per service based on existing compose files:
      - Cloud relay: 256M
      - Mail servers: 512M
      - Webmail (Roundcube): 256M, (SnappyMail): 128M
      - Rspamd: 256M, Redis: 64M
      - Caddy: 128M, Radicale: 128M
    - Include health checks matching existing Dockerfiles
    - Include deploy.resources.limits for all services
