**Create Radicale rights file (`home-device/caldav-carddav/radicale/rights`):**
This defines sharing permissions. Structure:

1. **Root collections access** — authenticated users can discover their own collections
2. **Owner full access** — users have full read/write to their own collections (user@domain/*)
3. **Shared collections** — all authenticated users can read/write shared family calendar and address book
4. **Principal access** — needed for CalDAV auto-discovery (PROPFIND on user principal)

```
# DarkPipe Radicale rights file
# Defines CalDAV/CardDAV sharing permissions
# Edit this file to customize calendar/contact sharing between household members

# Allow authenticated users to discover their own principal URL
[principal-discovery]
user: .+
collection:
permissions: R

# Owner has full access to their own collections
[owner-access]
user: .+
collection: {user}/.*
permissions: rw

# Allow users to create top-level collections in their namespace
[owner-root]
user: .+
collection: {user}
permissions: RW

# Shared family calendar and address book (all authenticated users)
[shared-family]
user: .+
collection: shared/.*
permissions: rw

# Shared collection root access
[shared-root]
user: .+
collection: shared
permissions: R
```

NOTE: The `{user}` token in Radicale rights files matches the authenticated username. This allows each user (e.g., alice@example.com) to access only their own collections (alice@example.com/*) while also accessing the shared/* collections.

**Create collection setup script (`home-device/caldav-carddav/setup-collections.sh`):**
Bash script that creates default calendar + address book for a user AND shared family collections.

Arguments: `setup-collections.sh <username> [--shared]`
- `<username>`: Create default personal calendar and address book for this user (e.g., alice@example.com)
- `--shared`: Also create shared family calendar and address book (run once on initial setup)

For personal collections, create:
- `/data/collections/collection-root/<username>/calendar.ics/` directory with `.Radicale.props` file setting `{DAV:}displayname: My Calendar` and `{urn:ietf:params:xml:ns:caldav}supported-calendar-component-set: VEVENT,VTODO`
- `/data/collections/collection-root/<username>/contacts.vcf/` directory with `.Radicale.props` file setting `{DAV:}displayname: My Contacts`

For shared collections (--shared flag), create:
- `/data/collections/collection-root/shared/family-calendar.ics/` with displayname "Family Calendar"
- `/data/collections/collection-root/shared/family-contacts.vcf/` with displayname "Family Contacts"

The `.Radicale.props` file is JSON format used by Radicale's multifilesystem storage:
```json
{"tag": "VCALENDAR", "{DAV:}displayname": "My Calendar", "{urn:ietf:params:xml:ns:caldav}supported-calendar-component-set": "VEVENT,VTODO,VJOURNAL"}
```
For address books:
```json
{"tag": "VADDRESSBOOK", "{DAV:}displayname": "My Contacts"}
```

Make the script executable (chmod +x).

- Accept `--user <user@domain>` flag to sync a single user
- Default to syncing all users
- After syncing users, call `setup-collections.sh <username>` for each new user
- Print instructions for creating shared collections on first run

**Modifications to cloud-relay/caddy/Caddyfile:**

Replace the placeholder CalDAV/CardDAV comments with actual configuration:

1. Add `redir /.well-known/caldav /radicale/ 301` — RFC 6764 CalDAV auto-discovery
2. Add `redir /.well-known/carddav /radicale/ 301` — RFC 6764 CardDAV auto-discovery
3. Add `reverse_proxy /radicale/* 10.0.0.2:5232` block with:
   - `header_up Host {host}`
   - `header_up X-Real-IP {remote_host}`
   - `header_up X-Forwarded-For {remote_host}`
   - `header_up X-Forwarded-Proto {scheme}`
   - Strip the /radicale prefix before forwarding: use `handle_path /radicale/*` and `reverse_proxy 10.0.0.2:5232`

The routing structure should be:
```
{$WEBMAIL_DOMAINS} {
    # CalDAV/CardDAV auto-discovery (RFC 6764)
    redir /.well-known/caldav /radicale/ 301
    redir /.well-known/carddav /radicale/ 301

    # CalDAV/CardDAV reverse proxy to home device Radicale
    handle_path /radicale/* {
        reverse_proxy 10.0.0.2:5232 {
            header_up Host {host}
            header_up X-Real-IP {remote_host}
            header_up X-Forwarded-For {remote_host}
            header_up X-Forwarded-Proto {scheme}
        }
    }

    # Webmail reverse proxy (default route)
    reverse_proxy 10.0.0.2:8080 {
        ... (existing config from 06-01)
    }
}
```

IMPORTANT: `handle_path` strips the `/radicale/` prefix before forwarding. This means Radicale receives requests at its root `/`, which is correct for its standard operation.

NOTE about Stalwart: When Stalwart is the mail server, CalDAV/CardDAV is built-in on port 8080 (same as webmail). Stalwart handles `/.well-known/caldav` and `/.well-known/carddav` internally. The Caddyfile well-known redirects and /radicale/ path only apply when Radicale is deployed (Maddy/Postfix+Dovecot profiles). Add a comment in the Caddyfile documenting this: "# CalDAV/CardDAV: Remove /radicale/* block when using Stalwart (built-in CalDAV/CardDAV on port 8080)"

For Stalwart users, the webmail reverse_proxy to 10.0.0.2:8080 already forwards CalDAV/CardDAV requests since Stalwart serves them on the same port. The well-known redirects should point to `/` instead of `/radicale/` for Stalwart. Add commented-out alternative lines:
```
# For Stalwart (built-in CalDAV/CardDAV), use these instead:
# redir /.well-known/caldav / 301
# redir /.well-known/carddav / 301
# (and remove the handle_path /radicale/* block)
```
