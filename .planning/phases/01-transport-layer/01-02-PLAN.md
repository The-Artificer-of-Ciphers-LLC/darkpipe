<objective>
Establish the mTLS alternative transport and internal PKI: step-ca deployment as private CA, mTLS server (cloud relay) and client (home device) with mutual certificate verification, persistent connection with exponential backoff reconnection, and automated certificate renewal via systemd timers.

Purpose: mTLS provides an application-level alternative to WireGuard for users who prefer it or cannot use kernel WireGuard. The internal CA (step-ca) issues short-lived certificates distinct from public-facing TLS, supporting both WireGuard cert needs and mTLS transport with automated renewal.

Output: Go packages for mTLS server/client with auto-reconnect, step-ca setup tooling, certificate renewal automation via systemd, deployment script for CA initialization.
</objective>

       - Function `InitCA(cfg CAConfig) error` that shells out to `step ca init` with the config values. This initializes the CA database, root cert, and intermediate cert. Returns error with clear message if step-ca binary not found.
       - Function `IssueCertificate(caURL, commonName, certPath, keyPath string) error` that shells out to `step ca certificate` to issue a cert from the CA. Used for initial cert issuance for both server and client.
       - Function `CheckCertExpiry(certPath string) (time.Time, error)` that reads the certificate and returns its NotAfter time. Uses crypto/x509 stdlib (no external deps).
       - All functions should wrap exec.Command with proper error handling and stderr capture for debugging.

    2. Create `transport/pki/ca/setup_test.go`:

       - Test CAConfig defaults are applied correctly.

    3. Create `deploy/pki/step-ca-setup.sh` (bash, executable):
       - Shebang with `set -euo pipefail`.
       - Check for root.
       - Download and install step-ca and step CLI if not present. Use Smallstep's official install method for the target architecture (detect amd64 vs arm64 with `uname -m`).
       - Run `step ca init --name "DarkPipe Internal CA" --dns "ca.darkpipe.internal" --address ":8443" --provisioner "darkpipe-acme" --provisioner-type ACME`.
       - Create systemd service for step-ca daemon.
       - Enable and start step-ca.
       - Install the cert-renewer systemd timer and service templates.
       - Print CA fingerprint and URL for client bootstrapping.

    4. Create `transport/pki/renewal/systemd/cert-renewer@.service`:
       ```ini
       [Unit]
       Description=Certificate renewal for %I
       After=network-online.target

       [Service]
       Type=oneshot
       ExecCondition=/usr/local/bin/step certificate needs-renewal /etc/darkpipe/certs/%i.crt
       ExecStart=/usr/local/bin/step ca renew --force /etc/darkpipe/certs/%i.crt /etc/darkpipe/certs/%i.key
       ExecStartPost=/bin/systemctl reload %i.service 2>/dev/null || true
       ```

    5. Create `transport/pki/renewal/systemd/cert-renewer@.timer`:
       ```ini
       [Unit]
       Description=Certificate renewal timer for %I

       [Timer]
       OnBootSec=5min
       OnUnitActiveSec=10min
       RandomizedDelaySec=5min

       [Install]
       WantedBy=timers.target
       ```

    6. Create `transport/pki/renewal/hooks/reload.sh` (executable):
       - Accepts service name as argument.
       - Runs `systemctl reload $1` to reload the service after cert renewal.
       - Logs action to journald via logger.

       - Struct `Server` with fields: listenAddr string, tlsConfig *tls.Config.
       - Function `NewServer(listenAddr, caCertPath, serverCertPath, serverKeyPath string) (*Server, error)`:
         - Load CA cert into x509.CertPool for client verification.
         - Load server cert/key pair via tls.LoadX509KeyPair.
         - Configure tls.Config with ClientAuth=tls.RequireAndVerifyClientCert (CRITICAL: enforces mutual auth), ClientCAs pointing to CA cert pool, MinVersion=tls.VersionTLS12.
         - Do NOT set cipher suites explicitly -- Go defaults include TLS 1.3 with post-quantum key exchange in Go 1.24+.
       - Method `Listen() (net.Listener, error)` that returns a tls.Listen listener.
       - Method `Serve(handler func(net.Conn))` that accepts connections in a loop and spawns goroutines for each.
       - Include proper error handling: log connection errors but don't crash the server.

    2. Create `transport/mtls/client/connector.go`:
       - Struct `Client` with fields: serverAddr string, tlsConfig *tls.Config.
       - Function `NewClient(serverAddr, caCertPath, clientCertPath, clientKeyPath string) (*Client, error)`:
         - Load CA cert for server verification (RootCAs).
         - Load client cert/key pair.
         - Configure tls.Config with MinVersion=tls.VersionTLS12.
       - Method `Connect(ctx context.Context) (*tls.Conn, error)` for single connection attempt.
       - Method `MaintainConnection(ctx context.Context, handler func(net.Conn) error) error`:
         - Use github.com/cenkalti/backoff/v4 for exponential backoff.
         - InitialInterval=1s, MaxInterval=5min, MaxElapsedTime=0 (never give up).
         - On successful connection, reset backoff.
         - On handler error or connection drop, retry with backoff.
         - Respect context cancellation for clean shutdown.
       - Run `go get github.com/cenkalti/backoff/v4` to add dependency.
