<task type="auto">
  <name>Task 1: Implement WireGuard health monitoring and unified transport health checker</name>
  <files>
    transport/wireguard/monitor/health.go
    transport/wireguard/monitor/health_test.go
    transport/wireguard/monitor/reconnect.sh
    transport/health/checker.go
    transport/health/checker_test.go
  </files>
  <action>
    1. Create `transport/wireguard/monitor/health.go`:
       - Use `golang.zx2c4.com/wireguard/wgctrl` library for kernel WireGuard control.
       - Function `CheckTunnelHealth(deviceName string, maxHandshakeAge time.Duration) error`:
         - Create wgctrl.Client, get device by name.
         - Check each peer's LastHandshakeTime.
         - Return error if handshake age exceeds maxHandshakeAge (default 5 minutes with PersistentKeepalive=25, handshakes refresh ~2min).
         - Return error if no peers configured.
         - Return nil if all peers healthy.
       - Function `Monitor(ctx context.Context, deviceName string, interval time.Duration, alertFn func(error))`:
         - Tick at interval (default 30s).
         - Call CheckTunnelHealth on each tick.
         - On failure: call alertFn with error details.
         - Respect context cancellation for clean shutdown.
       - Run `go get golang.zx2c4.com/wireguard/wgctrl` to add dependency.

    2. Create `transport/wireguard/monitor/reconnect.sh` (executable):
       - Checks WireGuard handshake age using `wg show wg0 latest-handshakes`.
       - If handshake age > 300s (5 min), restart the interface: `systemctl restart wg-quick@wg0`.
       - Log actions via logger for journald visibility.
       - Intended to be called by a systemd timer or cron as a fallback safety net alongside the Go monitor.

    3. Create `transport/health/checker.go`:
       - Type `TransportType string` with constants `WireGuard` and `MTLS`.
       - Struct `HealthStatus` with fields: Transport TransportType, Healthy bool, LastCheck time.Time, Error string, Details map[string]string.
       - Function `Check(transport TransportType, config interface{}) HealthStatus`:
         - For WireGuard: call wireguard/monitor.CheckTunnelHealth, populate Details with handshake age and peer count.
         - For mTLS: attempt TLS handshake to server address, check certificate validity, populate Details with cert expiry and connection latency.
       - This provides a unified health interface that Phase 9 (monitoring) will consume.

    4. Create tests:
       - `health_test.go`: Test CheckTunnelHealth returns error for non-existent device. Test Monitor calls alertFn when health check fails (mock or use build tag for integration). Skip wgctrl tests if not running as root (kernel access required).
       - `checker_test.go`: Test that Check returns correct TransportType. Test that unhealthy status includes error message. Test Details map is populated for both transport types.

    5. Run `go mod tidy`.
  </action>
  <verify>
    Run `go test ./transport/wireguard/monitor/...` and `go test ./transport/health/...` -- all tests pass (skip kernel-dependent tests gracefully). Run `go vet ./transport/...` -- no issues. Verify reconnect.sh passes `bash -n` syntax check.
  </verify>
  <done>
    WireGuard health monitoring checks handshake age and alerts on failures. Reconnection script restarts WireGuard if handshake stale. Unified health checker provides consistent interface for both WireGuard and mTLS transports. All tests pass.
  </done>

  <name>Task 2: Create integration test scripts and VPS provider documentation</name>
  <files>

  <action>
    1. Create `deploy/test/tunnel-test.sh` (executable):
       - Shebang with `set -euo pipefail`.
       - Accept transport type as argument: "wireguard" or "mtls".
       - For WireGuard:
         - Check wg0 interface exists (`ip link show wg0`).
         - Ping remote peer through tunnel (`ping -c 3 10.8.0.X`).
         - Check handshake age < 5 minutes (`wg show wg0 latest-handshakes`).
         - Transfer test data through tunnel (100KB via netcat or curl).
         - Report: latency, handshake age, data transfer success.
       - For mTLS:
         - Check that mTLS server port is listening (`ss -tln | grep 8443`).
         - Attempt TLS connection with client cert (`openssl s_client` or Go test binary).
         - Verify mutual authentication succeeded.
         - Report: connection status, cert validity, handshake time.
       - Exit 0 on all checks pass, exit 1 on any failure with clear error message.

    2. Create `deploy/test/outage-sim.sh` (executable):
       - Shebang with `set -euo pipefail`.
       - MUST run on home device (check hostname or accept confirmation).
       - Requires root (checks).
       - Steps:
         a. Verify tunnel is currently healthy (call tunnel-test.sh).
         b. Record pre-outage state (handshake time, peer status).
         c. Simulate outage: `systemctl stop wg-quick@wg0` (for WireGuard) or kill mTLS client process (for mTLS).
         d. Wait 60 seconds (per phase success criteria).
         e. Restore: `systemctl start wg-quick@wg0` (for WireGuard) or restart mTLS client.
         f. Wait up to 90 seconds for tunnel to re-establish, checking every 5 seconds.
         g. Verify tunnel is healthy again (call tunnel-test.sh).
         h. Report: time to recovery, handshake re-established, data flow resumed.
       - Exit 0 if auto-recovery succeeded, exit 1 if tunnel did not recover within 90 seconds.
       - This directly validates phase success criterion #3.

    3. Create `docs/vps-providers.md`:
       - Title: "VPS Provider Selection for DarkPipe"
       - Section: "Port 25 Requirement" -- explain why port 25 is critical (SMTP inbound/outbound).
       - Section: "Known Compatible Providers" with table:
         | Provider | Port 25 | Notes | Tested |
         | Vultr | Yes (after account verification) | May require support ticket for new accounts | No |
         | Hetzner | Yes | Available on all plans, good EU privacy laws | No |
         | OVH/OVHcloud | Yes | Must configure reverse DNS manually | No |
         | Linode/Akamai | Yes (after account age) | New accounts may have restrictions, contact support | No |
         | BuyVM | Yes | Known for SMTP-friendly policies | No |
       - Section: "Known Restricted Providers":
         | Provider | Port 25 | Notes |
         | DigitalOcean | Blocked by default | Must request unblock, often denied for new accounts |
         | AWS EC2 | Blocked by default | Must request removal of port 25 throttle via support |
         | Google Cloud | Blocked | Cannot send on port 25, period |
         | Azure | Blocked | Enterprise accounts only |
       - Section: "How to Validate Your Provider" with step-by-step:
         1. Provision smallest VPS (typically $3-5/month).
         2. Test inbound: `nc -l 25` on VPS, telnet from external.
         3. Test outbound: `telnet gmail-smtp-in.l.google.com 25` from VPS.
         4. Check reverse DNS: provider must allow PTR record configuration.
         5. If any test fails, contact provider support or choose a different provider.
       - Section: "Minimum VPS Specs" -- 1 vCPU, 512MB RAM, 10GB SSD, static IPv4, PTR record support.
       - Mark all provider entries as "Tested: No" -- these will be validated in actual deployment.
